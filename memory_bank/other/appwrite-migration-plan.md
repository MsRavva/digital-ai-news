# Appwrite Migration Plan

## Цель

Подготовить безопасный поэтапный перенос текущего проекта с `Supabase` на `Appwrite` без потери пользователей, ролей, публикаций и рабочего auth-flow.

## Текущее использование Supabase

- `Supabase Auth` используется для email/password и OAuth.
- Серверные и клиентские auth helpers завязаны на `@supabase/ssr` и `@supabase/supabase-js`.
- Таблица `profiles` является источником ролей `student | teacher | admin`.
- Контентные сущности завязаны на Supabase-таблицы: `posts`, `tags`, `post_tags`, `comments`, `likes`, `views`.
- В проекте есть Supabase-специфичная логика callback flow, server-side guard и профилирования пользователя после OAuth.
- В проекте накоплена инфраструктурная SQL-логика (`supabase/*.sql`), включая миграции и серверные функции.

## Ключевой вывод

Это не только миграция базы данных. Это миграция одновременно:

- auth provider;
- модели профилей и ролей;
- data access layer;
- серверного session flow;
- части административной логики.

Рекомендуемый путь: поэтапная миграция через слой абстракции, а не один большой cutover без промежуточной стабилизации.

## Целевая модель Appwrite

### Auth

- Использовать `Appwrite Account` как канонический источник identity.
- Канонический идентификатор пользователя: `account.$id`.
- Не хранить права доступа только в клиентских preferences.

### Данные

Минимальный набор коллекций:

- `profiles`
- `posts`
- `tags`
- `post_tags` или relation-модель между постами и тегами
- `comments`
- `likes`
- `views`
- при необходимости отдельная коллекция для технических событий auth

### Роли

- Роль хранить в отдельной server-managed коллекции `profiles`.
- Все чувствительные проверки прав оставлять на сервере.

## Основные риски

### 1. Переписывание auth-flow

- Текущий flow уже стабилизирован и завязан на Supabase session model.
- Наиболее рискованные файлы: `middleware.ts`, `lib/auth-server.ts`, `context/auth-context.tsx`, `app/auth/callback/route.ts`, `lib/supabase-auth.ts`.

### 2. Смена модели идентификаторов

- Текущий проект использует Supabase user id как `profiles.id` и как внешний ключ в контентных сущностях.
- В Appwrite user ids будут другими, поэтому понадобится явный mapping старых Supabase IDs к новым Appwrite IDs.

### 3. Реляционная модель

- `post_tags`, `likes`, `views`, `comments` и авторские связи в Postgres richer, чем типичный document access pattern.
- Нужно заранее решить, использовать relation attributes Appwrite или хранить string reference + серверную агрегацию.

### 4. Legacy SQL-логика не переносится напрямую

- SQL triggers, backfill-сценарии и служебные миграции нельзя переносить в Appwrite как есть.
- Их нужно заменить server-side orchestration и/или Appwrite Functions.

## Рекомендуемая стратегия миграции

## Этап 1. Зафиксировать целевую схему Appwrite

Нужно заранее определить:

- какие коллекции создаются;
- какие поля обязательны;
- какие индексы нужны;
- как моделируются связи;
- где хранится роль;
- как проверяется доступ на сервере.

Результат этапа:

- утвержденная целевая схема данных Appwrite;
- утвержденная модель ролей и permissions.

## Этап 2. Вынести provider-agnostic слой

До реальной миграции нужно отвязать UI и route handlers от прямых вызовов Supabase.

Нужно выделить сервисы:

- `auth service`
- `profile service`
- `posts service`
- `comments service`
- `admin service`

Цель:

- UI и pages зависят от внутренних сервисов проекта, а не от `lib/supabase-*` напрямую.

## Этап 3. Миграция чтения данных

Переносить сначала безопасные read-сценарии:

- главная страница;
- архив;
- страница публикации;
- чтение тегов и категорий;
- чтение комментариев.

Причина:

- это позволяет проверить Appwrite schema и индексы без риска сломать login flow.

## Этап 4. Миграция профилей и ролей

Подготовить отдельную миграцию:

- экспорт всех `profiles` из Supabase;
- создание коллекции `profiles` в Appwrite;
- загрузка ролей, username, email и служебного mapping.

Обязателен mapping:

- `legacySupabaseUserId`
- `appwriteUserId`
- `email`
- `username`
- `role`

## Этап 5. Выбрать стратегию переноса auth

### Вариант A. Forced relink / повторный вход

- пользователи заново входят через email/password или OAuth;
- при первом входе выполняется безопасная серверная привязка профиля по email.

Плюсы:

- проще и надежнее.

Минусы:

- хуже UX в момент миграции.

### Вариант B. Серверный импорт пользователей

- пользователи создаются в Appwrite административно;
- затем профили перепривязываются к новым account IDs.

Плюсы:

- мягче для пользователей.

Минусы:

- выше сложность и риск.

Рекомендация:

- сначала проверить возможности Appwrite для controlled import;
- если полный бесшовный импорт ненадежен, использовать гибридную стратегию.

## Этап 6. Переписать auth-flow на Appwrite

После подготовки profile model и mapping:

- переписать `context/auth-context.tsx`;
- переписать `middleware.ts`;
- переписать `lib/auth-server.ts`;
- заменить `lib/supabase-auth.ts`;
- переписать `/login`, `/register`, `/auth/callback`, `/auth/post-login`.

При этом обязательно сохранить:

- безопасный redirect после логина;
- server-side guard;
- единое поведение login/register/OAuth.

## Этап 7. Перенести write-сценарии

После стабилизации auth:

- создание постов;
- редактирование постов;
- комментарии;
- лайки;
- просмотры;
- обновление профиля;
- административные операции.

## Этап 8. Cutover

Порядок:

1. заморозить записи в Supabase;
2. выгрузить финальную дельту данных;
3. загрузить финальную дельту в Appwrite;
4. переключить окружение проекта на Appwrite;
5. выполнить smoke-test;
6. оставить короткий rollback window.

## Этап 9. Удаление Supabase legacy

После успешной стабилизации:

- удалить `lib/supabase-*`;
- удалить неактуальные `supabase/*.sql`;
- удалить Supabase env variables;
- удалить Supabase-зависимости;
- обновить документацию и memory bank.

## Практический порядок работ

1. Спроектировать Appwrite schema и permissions.
2. Вынести provider-agnostic слой.
3. Перенести read-path.
4. Перенести profile и role model.
5. Перенести auth.
6. Перенести write-path.
7. Выполнить cutover.
8. Удалить Supabase legacy.

## Что не делать

- Не делать one-shot миграцию без промежуточной стабилизации.
- Не переносить Supabase SQL-триггеры один-в-один.
- Не хранить роли только в клиентских настройках пользователя.
- Не начинать с переписывания всего UI до проектирования схемы данных и auth strategy.
